Harness 工程迭代：多 Agent 系统从 Demo 到生产的 10 轮进化

总览

overview

从工程实现视角，把多 Agent 金融研究系统拆成目标、机制选择、上下文约束和稳定性策略。

“Harness 不是外壳，而是让 Agent 长流程可控的工程骨架。”

这是什么

这是一份从基础 Agent Loop 出发的工程化迭代方案。它用一个 A 股研究报告系统串起 10 轮 Harness 迭代，展示如何把"能跑"的多 Agent Demo，逐步改造成可控、可扩展、可恢复、可交接上下文的系统规格。

Harness 是什么

Model + Harness = Agent。模型本身（训练、推理、参数优化）不是 Harness。模型之外的一切——怎么给模型喂信息、怎么管理工具、怎么设计交互、怎么处理错误、怎么让 Agent 在真实环境中跑起来——才是 Harness。

如果把 Agent 比作自动驾驶汽车，模型是发动机，那 Harness 就是底盘、转向、刹车、传感器融合、路径规划。发动机再强，没有这些东西，车也开不起来。

类别组件解决的问题
工具层工具分发、安全沙箱、权限管线、Hooks工具怎么安全地接入和扩展
协作层规划管理、Skill 加载、上下文压缩多 Agent 怎么高效协作
记忆层记忆系统、Prompt 运行时组装知识怎么跨会话保留
韧性层错误恢复、子 Agent 隔离系统怎么不崩溃

核心洞察

  • Harness 决定 Agent 能走多远：模型能力需要循环、工具、权限与状态系统承接。
  • 多 Agent 协作需要显式的上下文隔离，不能让执行细节污染调度上下文。
  • 工具不是越多越好，分发映射、沙箱、权限管线和 Hook 才让工具可控。
  • 长流程可靠性来自规划、压缩、记忆、运行时 Prompt 与错误恢复的组合。
